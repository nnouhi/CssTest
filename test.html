Nielsen's Heuristics Compliance Assessment

  1. Visibility of system status

    1. Headings on top of every page so that the user knows what page he/she currently is.

    2. Every page has the same navigation buttons to help the user navigate through all available pages.

    3. Buttons change colour when they are being pressed or hovered.

    4. Loading message is displayed where is needed to let the user know that the page is still running.

    5. If a session is not finished alert messages ask the user if he/she wants to continue the unfinished session.

  2. Match between system and the real world

    1. All pop ups/alerts use words, phrases, and concepts familiar to the user; avoided using programming/technical language.

    2. Most of the buttons have an icon that describes their usage.

  3. User control and freedom

    1. Returning user can start a new game or continue where he left off.

    2. If the question allows it, a user can skip an unwanted question and move the next one.

    3. When pressing the skip button a pop up appears to ask the user if he/she wants to actually skip the question (Avoid unwanted actions)

    4. User can activate the QR code scanner and then change to his camera of choice (front/back)

    5. User can copy the scanned message with a copy to clipboard button

  4. Consistency and standards

    1. All pages follow the same design template.

    2. All pages follow the same structure template.

  5. Error prevention

    1. If an answer requires an Integer value and the user submits a float an alert message appears and prompts the user to enter an Integer.

    2. A message appears when the user submits an 'empty' answer.

    3. A message is displayed to alert user if user device lacks a feature needed in the app. (No cameras available for QR Scanner ft).
    4. A message is displayed to alert user if user wants to play with a username that was used before.

  6. Recognition rather than recall

    1. All pages follow the same template; user gets familiar with the app (no recall).

    2. Button's use is clear; user doesn't need to guess what does what (no recall).

    3. Minamalist UI and app design ; user doesn't need to guess what does what (no recall).

  7. Flexibility and efficiency of use

    1. App can be used by an experienced and also by an inexperienced user.

  8. Aesthetic and minimalist design

    1. All pages follow an aesthetic and minimalist design.

    2. Interfaces do not contain information which is irrelevant or rarely needed.

    3. Minamalist UI and app design ; user doesn't need to guess what does what (no recall).

    4. Button icons used to avoid any recall and avoid unnecessary information.

  9. Help users recognize, diagnose, and recover from errors

    1. A message appears every time that an error occurs; User can restart the session any time an error occurs.

    2. All error messages use every day language so that the user can understand what error occured; avoided using programming/technical language.

  10. Help and documentation

    1. All error messages use every day language so that the user can understand what error occured; avoided using programming/technical language.

    2. All pages include an 'info' button at the top that gives a brief explanation on how to play the game.

User Acceptance Test

  1. The app will be tested mostly by George Antoniou, family members and by some of our friends.

  2. All test made will be documented by Lev Isaev.

  3. The testing will begin when the functional part of the app is finished and only the styling needs to be done and the app is close to being relased.

  4. Every functional part of the app that was implemented will be thoroughly tested.

  5. All testing results are going to be documented in the User Acceptance Test page (linked from landing page).

  6. Discovered bugs within the app will be reported to Lev Isaev and then reported to Nicolas Nouhi, Nicolas Nouhi then will try to come up with the appropriate fix.

  7. The app will be test twice. Once when app's functionality is implemented, and twice when the bugs of the first testing are fixed.

User Acceptance Test cases

Test Test Type Description
1 Alpha & Beta Testing Buttons (Submit) -> When user clicks on any button move to the next section.
2 Alpha & Beta Testing Username (player name) -> When user provides a new name user must be able to view available challenges. If username was used to play any treasure hunt before, alert the user that the name was used. If a user was playing an unfinished session alert the user if user wants to restart with a new name or continue the session.
3 Alpha & Beta Testing Skip Button -> Skip button should be visible only if the question can be skipped. Also when button is clicked; A pop should ask the user if user wants to confirm the skip action.
4 Alpha & Beta Testing TH Challenges User should be able to play any TH Challenge that is available at the given time. If a TH Challenge is not available at the moment display a message to the user.
5 Alpha & Beta Testing QR Code reader (Switch Camera / Toggle Camera ) buttons -> User should be able to toggle camera on/off whenever user wants, user should also be able to choose whatever device camera he/she prefers. If device cameras are not found user will receive an error message
6 Alpha & Beta Testing QR Code decoded message -> User should be able to view the decoded message
7 Alpha & Beta Testing Copy to clipboard button -> User should be able to copy the decoded message by pressing the copy to clipboard button; When message is copied display an alert that informs the user of the action
8 Alpha & Beta Testing Current question message -> User should be able to view the current question and how many are left
9 Alpha & Beta Testing Current player score -> User should be able to view the current score
10 Alpha & Beta Testing Points gained / lost after answering a question -> User should be able to view how many points user won or lost after asnwering a question
11 Alpha & Beta Testing Answering mechanism -> User's answers should be saved and compared with the correct answers; detract points if answer is wrong , add points if answer is correct
12 Alpha & Beta Testing Location -> Update location periodically every i.e. 30 seconds regardless of the user answering questions. Update the location when receiving the question, but right before the answer is submitted.
13 Alpha & Beta Testing Leaderboards -> After all questions are anwered display a message that informs the user that the session is over, Load leaderboards and display the user's score and also user's ranking relative to other players who played before. If player is in top ranking highlight users row within the leaderboard.
14 Alpha & Beta Testing Responsive design -> All pages must be responsive to all devices.
